# WordGameByAI 项目现状

> 状态快照：2026-07-10。本文以当前 `main` 工作区中的代码、配置和测试为准。

## 1. 当前阶段

WordGameByAI 已完成一个可本地运行的全栈 AI 文字 RPG 原型。项目具备完整的前后端目录、数据库模型、OpenAI-compatible 大模型调用、多 Agent 剧情编排、动态记忆检索和存档管理能力，适合继续用于互动叙事 Demo、课程展示和产品原型验证。

当前更接近“功能完整的开发版”，尚不是开箱即用的公网生产服务。

## 2. 已实现能力

| 领域 | 当前实现 |
| --- | --- |
| 用户与权限 | 注册、登录、验证码、Bearer Token、管理员和普通用户数据边界 |

## 3. 运行边界

- 后端默认使用本地 SQLite；公网部署文档建议切换到 MySQL。
- AI 能力依赖 OpenAI-compatible Chat Completions 服务；未配置 API Key 时应用可启动，但无法完成 Agent 调用。
- 当前 RAG 使用项目内实现的哈希向量，不依赖外部 Embedding 服务，适合轻量原型，不等同于生产级语义检索。
- 上传头像和模型私有配置默认写入本地文件系统；多实例部署前需要改为共享对象存储或集中配置服务。
- 限流和登录态适用于当前应用形态；正式公网运营仍需补充网关级限流、监控、审计和密钥轮换。

## 4. 测试与工程现状

- 2026-07-10 使用项目 `.venv` 执行后端测试：37 项全部通过；运行中有 `datetime.utcnow()` 弃用警告。
- 同日执行 `npm --prefix frontend run build`：Vue/Vite 生产构建通过。
- `tests/test_dynamic_rag.py` 覆盖动态记忆、快速模式、状态补丁和结构化输入等后端核心行为。
- 前端提供 `npm run build`，当前没有独立的单元测试或端到端测试目录。
- 仓库未配置 GitHub Actions、Docker Compose 或数据库迁移框架。
- `.gitignore` 已排除真实环境变量、模型私有配置、SQLite 数据库、上传文件、依赖和构建产物。

## 5. 建议优先级

1. 增加 GitHub Actions，至少执行后端测试和前端生产构建。
2. 为登录、存档导入导出和流式剧情补充 API 集成测试。
3. 增加前端 E2E，覆盖注册、创建存档、生成开场和推进回合主流程。
4. 引入数据库迁移工具，并明确 SQLite 与 MySQL 的版本升级策略。
5. 公网部署时接入 HTTPS、对象存储、集中日志、备份和可观测性。
