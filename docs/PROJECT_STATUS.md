# WordGameByAI 项目现状

> 状态快照：2026-07-17。本文以当前工作区中的代码、配置和测试为准。

## 1. 当前阶段

WordGameByAI 已形成包含 Vue 网页端、Flutter Android 客户端和版本化移动 API 的全栈 AI 文字 RPG。Android 已完成模拟器公网登录、建档、AI 开场和主要工作区联调，不再只是客户端骨架。

当前更接近“功能完整的开发版”，尚不是开箱即用的公网生产服务。

## 3. 运行边界

- 后端默认使用本地 SQLite；公网部署文档建议切换到 MySQL。
- AI 能力依赖 OpenAI-compatible Chat Completions 服务；未配置 API Key 时应用可启动，但无法完成 Agent 调用。
- 当前 RAG 使用项目内实现的哈希向量，不依赖外部 Embedding 服务，适合轻量原型，不等同于生产级语义检索。
- 上传头像和模型私有配置默认写入本地文件系统；多实例部署前需要改为共享对象存储或集中配置服务。
- 应用层限流和登录态已持久化；正式公网运营仍需补充网关级限流、监控、审计和密钥轮换。

## 4. 测试与工程现状

- 当前测试声明：Python 后端 46 项、Flutter 15 项、Vue/Node 5 项，共 66 项。
- 最近一次完整验证执行了后端测试、Flutter 测试与静态分析、Vue/Node 测试及 Vite 生产构建。
- 后端测试覆盖动态记忆、快速模式、状态补丁、结构化输入、移动分页、事务回滚、并发租约、认证轮换和输出质量诊断。
- 前端已有 5 项默认头像资源与匹配测试，但仍没有组件测试或端到端测试。
- 仓库未配置 GitHub Actions、Docker Compose 或正式数据库迁移框架；当前启动迁移能兼容本次模型增量，但更复杂升级仍应引入 Alembic。
- `.gitignore` 已排除真实环境变量、模型私有配置、SQLite 数据库、上传文件、依赖和构建产物。

## 5. 建议优先级

1. 增加 GitHub Actions，至少执行后端测试和前端生产构建。
2. 增加前端与 Flutter E2E，覆盖登录、创建存档、生成开场和推进回合主流程。
3. 将 Android 改为唯一正式包名和 release keystore，强制 HTTPS 后完成商店发行验收。
4. 引入 Alembic，明确 SQLite 与 MySQL 的版本升级策略。
5. 公网部署时接入对象存储、集中日志、备份和可观测性。
